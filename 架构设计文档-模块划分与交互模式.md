> **定位**：本文档定义技术中台的物理模块结构、分层职责，以及"客户端SDK + 独立服务端"的核心交互模式。  
> **目标读者**：架构师、Tech Lead、核心开发。  
> **配套文档**：《技术中台建设规范白皮书 (终极执行版)》

## 1. 总体项目结构

```

├── trust-cloud-parent/                   # 【父POM】定义插件、编译规则、通用配置

├── trust-common-core/                    # 基础内核 (纯工具，无服务端)
├── trust-common-config/                  # 配置中心客户端封装 (纯客户端，无独立服务端)
├── trust-common-db/                      # 数据库持久层封装 (纯工具，无服务端)
├── trust-common-log/                     # 日志规范与链路追踪 (纯工具，无服务端)
├── trust-common-web/                     # Web层通用封装 (纯工具，无服务端)
│
├── trust-common-security/                # 【客户端】安全与权限SDK
├── trust-common-file/                    # 【客户端】文件服务SDK
├── trust-common-mq/                      # 【客户端】消息队列SDK

├── trust-common-rpc/                     # 【客户端】RPC通信SDK

├── trust-auth-service/                   # 【服务端】统一鉴权与权限中心  (待创建)
├── trust-file-service/                   # 【服务端】文件存储管理中心      (待创建)
├── trust-audit-service/                  # 【服务端】审计日志中心          (待创建)
├── trust-gateway-service/                # 【服务端】API网关 (已存在)
│
├── trust-quant-account-service/          # 【业务服务】量化账户服务 (已存在)
├── trust-quant-quotation-service/        # 【业务服务】量化行情服务 (新增)
├── trust-quant-trade-service/            # 【业务服务】量化交易服务 (已存在)
└── ...                                   # 其他业务微服务
```

## 2. 三层架构定义

### 2.1 第一层：工程骨架层

| 模块 | 类型 | 职责 |
|------|------|------|
| `trust-cloud-parent` | Parent POM | 定义 Maven 插件版本、Java 编译版本、编码规范、通用 Profile（dev/test/prod）、子模块聚合。所有业务服务必须继承此 Parent。**不管理依赖版本号**。 |
| `trust-cloud-bom` | BOM POM | 统一管理所有外部依赖（Spring Boot、PostgreSQL Driver、Redisson、Flowable 等）及内部 Common 包的版本号。供 Parent 导入，也支持外部项目按需引用。**是版本仲裁的唯一真理源**。 |

**设计意图**：
- Parent 管"怎么编译"，BOM 管"用什么版本"。解耦构建规则与依赖清单。
- 当需要升级 Spring Boot 3.2 → 3.3 时，只改 BOM 一处，全量服务无痛升级。

### 2.2 第二层：基础组件层 (Common)

Common 模块分为两类：

#### A. 纯工具类组件 (无对应服务端)

> 这类组件提供**本地能力**，被业务服务直接依赖后，在进程内完成工作。**不需要网络调用**。

| 模块 | 核心能力 | 使用方式 |
|------|----------|----------|
| `trust-common-core` | BigDecimal→String Jackson 序列化、全局异常基类（BusinessException）、Snowflake ID 生成器、通用常量、工具类 | 所有服务直接依赖 |
| `trust-common-config` | Apollo 客户端封装、环境配置热刷新监听、加密配置解密（Jasypt） | 所有服务直接依赖 |
| `trust-common-db` | MyBatis Plus 配置、Flyway 迁移、分页插件、BlockAttack 拦截器、多租户拦截器 | 需要访问 DB 的服务依赖 |
| `trust-common-log` | TraceID 生成与传递（MDC）、JSON 格式日志模板、SkyWalking/Sleuth 适配 | 所有服务直接依赖 |
| `trust-common-web` | 统一响应体 `R<T>`、XSS 过滤器、防重放拦截器、BaseRequest 基类 | 对外暴露 HTTP 接口的服务依赖 |

#### B. 客户端SDK组件 (有对应独立服务端)

> 这类组件是**服务端的"驱动程序"**。业务服务依赖客户端 JAR，通过客户端内置的调用逻辑，**远程访问独立服务端**。
> 
> **设计原则**：业务服务不直接调用服务端 HTTP/Dubbo 接口，必须通过客户端 SDK 封装。

| 客户端模块 | 客户端职责 (SDK) | 对应服务端 | 服务端职责 |
|-----------|------------------|-----------|-----------|
| `trust-common-security` | 提供 `@DataScope` 注解（自动拼接权限 SQL）、Token 解析工具、RBAC-D 模型常量、Feign/Dubbo 调用客户端 | `trust-auth-service` | 用户/角色/菜单管理、Token 签发与校验、数据权限规则计算、部门层级查询 |
| `trust-common-file` | 提供 `FileClient.upload()/download()` API、文件流封装、分片上传工具 | `trust-file-service` | 文件元数据管理、存储策略（本地/OSS/MinIO）、文件清理调度 |
| `trust-common-mq` | 提供 `ReliableProducer`（本地消息表模式）、消息体序列化、消费端注解封装 | `trust-message-service` (可选) | 消息轨迹查询、死信队列管理、消息补发调度 |
| `trust-common-redis` | 提供 `@DistLock` 分布式锁、`@Cache` 注解、Redisson 客户端配置、限流工具 | Redis 集群 (基础设施) | —— (Redis 为中间件，非自研微服务) |
| `trust-common-rpc` | 提供服务发现适配（Spring Cloud Kubernetes）、负载均衡配置、熔断降级模板 | —— (由 K8s + Spring Cloud 提供) | —— |

**说明**：
- `trust-common-redis` 和 `trust-common-rpc` 的"服务端"是基础设施（Redis / K8s），因此严格来说是**中间件客户端封装**，不是微服务配对。
- 真正的"客户端-微服务配对"是 **security → auth-service** 和 **file → file-service**。
- `trust-common-mq` 可独立存在，也可合并进一个 `trust-message-service`。

### 2.3 第三层：业务服务层

| 已有服务 | 职责 |
|---------|------|
| `trust-gateway-service` | 统一入口、路由转发、鉴权前置（调用 auth-service）、限流熔断 |
| `trust-quant-account-service` | 量化账户开户、资产查询、费用计算、持仓查询 |
| `trust-quant-quotation-service` | 行情数据订阅、实时报价推送、历史行情查询、标的快照 |
| `trust-quant-trade-service` | 交易指令接收、委托下单、交易校验、持仓管理。**作为交易编排核心，聚合调用 account 与 quotation 服务** |

**业务服务调用关系**：

```
┌─────────────────────────────────────────┐
│  trust-quant-trade-service              │
│  (交易编排层 / 聚合服务)                  │
│                                         │
│  1. 下单前校验账户状态                    │
│     └── Feign调用 ──▶ trust-quant-account-service │
│                        (查询资产/持仓/冻结金额)      │
│                                         │
│  2. 获取实时行情计算委托价格              │
│     └── Feign调用 ──▶ trust-quant-quotation-service │
│                        (查询标的最新价/涨跌停价)     │
│                                         │
│  3. 本地执行交易规则校验                  │
│     └── LiteFlow 规则链 (本地 or 调用 rule-service) │
│                                         │
│  4. 下单后更新账户持仓                    │
│     └── MQ消息 ────▶ trust-quant-account-service │
│                        (异步更新持仓/资产)          │
└─────────────────────────────────────────┘
```

> **设计说明**：
> - `trust-quant-trade-service` 是**交易域的编排入口**，本身不保存账户资产和行情数据，通过 RPC 调用专门的服务获取。
> - 读操作（查账户、查行情）采用 **同步 Feign 调用**（超时配置 3s，熔断降级）。
> - 写操作（更新持仓）采用 **异步消息解耦**（通过 `trust-common-mq` 发送，确保最终一致性）。

| 建议补齐的服务 | 职责 |
|---------------|------|
| `trust-auth-service` | 统一身份认证与授权中心 |
| `trust-file-service` | 文件全生命周期管理 |
| `trust-audit-service` | 审计日志收集、Diff 查询、归档 |
| `trust-bpm-service` | Flowable 流程引擎托管、流程定义管理 |
| `trust-rule-service` | LiteFlow 规则链托管、规则热刷新 |

## 3. "客户端-服务端"模式详解

### 3.1 为什么采用这种模式？

在金融技术中台中，我们遇到以下痛点：

| 痛点 | 传统做法的问题 | 客户端-服务端模式的解决 |
|------|--------------|----------------------|
| **权限逻辑散落在各服务** | 每个服务自己写权限判断，规则不一致 | 统一收拢到 `trust-auth-service`，各服务通过 `trust-common-security` 客户端标准化接入 |
| **鉴权协议变更成本高** | 从 Session 改 JWT，所有服务都要改 | 只改客户端 SDK 和服务端，业务服务无感知 |
| **文件存储策略多变** | 本地磁盘 → OSS → 信创存储，业务代码侵入深 | 业务只调用 `FileClient`，存储策略由 `trust-file-service` 控制 |
| **审计数据分散难追溯** | 各服务自己写审计表，格式不统一 | `trust-audit-service` 统一收集，客户端 SDK 负责切面拦截 |

**本质**：把"怎么做"（协议、调用、序列化）封装进客户端 SDK，把"做什么"（权限计算、文件存储、日志归档）收拢到独立服务。

### 3.2 调用链路示例

#### 场景：业务服务查询"本部门数据"

```
┌─────────────────────────────────────────────────────────────┐
│  trust-quant-account-service (业务服务)                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Controller 层                                       │   │
│  │  @DataScope(deptField = "dept_id")                  │   │
│  │  public R<List<Account>> list(AccountQuery q) {...} │   │
│  └─────────────────────────────────────────────────────┘   │
│                          │                                  │
│  依赖 trust-common-security (客户端SDK)                      │
│  依赖 trust-common-db     (MyBatis拦截器)                    │
│                          ▼                                  │
│  MyBatis 拦截器自动读取当前用户部门列表                        │
│  自动拼接 SQL: WHERE dept_id IN (10, 11, 12)               │
│                          │                                  │
└──────────────────────────┼──────────────────────────────────┘
                           │
         ┌─────────────────┘
         │  (本地进程内完成，无需远程调用)
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  若需查询"用户拥有哪些数据权限"                              │
│  (如：用户能看见哪些部门的账户？)                             │
│                                                             │
│  trust-common-security 内置 Feign Client                    │
│  调用 trust-auth-service: GET /api/scope/depts/{userId}    │
│                          │                                  │
└──────────────────────────┼──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  trust-auth-service (独立服务端)                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  查询 RBAC-D 模型：用户→角色→数据范围→部门列表          │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**关键点**：
- `@DataScope` 的**拦截和 SQL 拼接**在本地完成（性能最优）。
- 权限规则的**计算**（用户能看哪些部门）由 `trust-auth-service` 远程提供（逻辑统一）。
- 业务服务**零权限逻辑代码**，只加一个注解。

### 3.3 客户端SDK设计规范

所有客户端 SDK 必须遵循以下结构：

```
trust-common-xxx/
├── src/main/java/com/trust/common/xxx/
│   ├── client/                 # Feign/Dubbo 远程调用接口 (对应服务端)
│   ├── annotation/             # 供业务服务使用的注解
│   ├── interceptor/            # 自动生效的拦截器 (如 MyBatis 拦截器)
│   ├── config/                 # XxxAutoConfiguration (Spring Boot 自动配置)
│   └── dto/                    # 客户端与服务端交互的 DTO
├── src/main/resources/
│   └── META-INF/spring.factories   # 自动装配入口
└── pom.xml
```

**自动装配约定**：
- 客户端 SDK 必须基于 Spring Boot Starter 机制，业务服务引入依赖后能力**自动生效**。
- 配置项统一前缀：`trust.xxx.*`（如 `trust.security.enabled=true`）。

## 4. 依赖关系规范

### 4.1 禁止的依赖（红线）

```
❌ 业务服务 直接依赖 Spring Cloud Alibaba Nacos
❌ 业务服务 直接操作 K8s API
❌ 业务服务 直接使用原生 JDBC / RedisTemplate / RabbitMQ Channel
❌ 业务服务 直接调用 外部系统 (恒生O45 / 融选) —— 必须通过 ACL Gateway
❌ 客户端SDK 之间循环依赖
```

### 4.2 推荐的依赖方向

```
trust-cloud-parent (所有服务继承)
    └── import trust-cloud-bom (版本管理)

业务服务 (如 trust-quant-account-service)
    ├── trust-common-core
    ├── trust-common-config
    ├── trust-common-db        (需要DB时)
    ├── trust-common-log
    ├── trust-common-web       (暴露HTTP接口时)
    ├── trust-common-security  (需要权限控制时)
    ├── trust-common-file      (需要文件时)
    └── trust-common-mq        (需要消息时)

客户端SDK (如 trust-common-security)
    ├── trust-common-core      (依赖工具类)
    └── trust-common-rpc       (依赖通信封装)
    └── (可选) trust-common-log
```

## 6. 总结

本架构通过三层划分，实现了以下目标：

1. **标准化**：Parent + BOM 统一工程骨架与版本，消除"各自为政"。
2. **工业化**：纯工具组件提供本地能力，客户端-服务端组件提供分布式能力，业务服务只关注业务。
3. **可演进**：权限协议、文件存储、消息格式变更时，只需升级客户端 SDK 和服务端，业务服务无侵入。
4. **金融级**：审计、权限、数据精度等合规能力下沉到基础设施，不依赖开发人员自觉。
