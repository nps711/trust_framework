

**版本**：V1.0 (Final Release)  
**适用范围**：固收业务系统、量化交易系统、综合运营管理平台  
**核心目标**：通过“工业化”的技术封装，构建一套“外骨骼”体系，让初中级开发人员只需关注业务逻辑，降低认知门槛，同时强制保障系统的金融级稳定性与合规性。

## 1. 总体架构设计原则
### 1.1 核心理念
+ **屏蔽底层 (Shielding)**：业务开发严禁直接操作 K8s API、原生 JDBC 或原始中间件 SDK，必须通过中台封装包 (`trust-common-*`) 调用。
+ **物理隔离 (Isolation)**：核心交易与外部系统（恒生 O45、融选、估值）必须通过防腐层 (ACL) 进行物理与逻辑双重隔离。
+ **约定大于配置 (Convention)**：强制规定工程结构、错误码、交互方式，消除“个人艺术风格”的代码。
+ **合规铁律 (Compliance)**：审计日志、数据版本管理、权限隔离是系统的生命线，不可妥协。

### 1.2 技术栈选型 (BOM)
基于 **K8s Native** 与 **信创国产化** 趋势的精简架构：

| 领域 | 选型方案 | 关键决策说明 |
| :--- | :--- | :--- |
| **开发语言** | Java 17 / 21 (LTS) | 兼顾生态与 ZGC 低延迟性能（量化场景必需）。 |
| **基础框架** | Spring Boot 3.x | 行业标准。 |
| **微服务底座** | **Spring Cloud Kubernetes** | **去 Nacos 注册中心**。直接利用 K8s Service (DNS) 做服务发现，架构极简，免维护。 |
| **配置中心** | **Apollo (携程)** | **保留**。相比 ConfigMap，提供细粒度权限管控、热发布和版本回滚，适合金融场景。 |
| **网关** | Spring Cloud Gateway | 统一入口，负责限流、熔断、鉴权。 |
| **数据库** | **PostgreSQL 15+** | **替代 MySQL**。利用其 JSONB 存储非结构化表单，强大的计算能力适配未来 GaussDB 迁移。 |
| **持久层** | MyBatis Plus | 国产标准，插件体系丰富（用于数据权限、多租户）。 |
| **任务调度** | **XXL-JOB** | **保留**。轻量级分布式调度，用于日终跑批。 |
| **流程引擎** | **Flowable 6.x** | 嵌入式工作流，用于运营审批。 |
| **表单引擎** | **formily** | 中后台复杂场景的表单解决方案，它也是一个表单框架 |
| **规则引擎** | **LiteFlow** | 国产轻量级编排引擎，用于风控准入、费用计算。 |
| **中间件** | RabbitMQ, Redis (Redisson) | 消息解耦与分布式锁标准件。 |

## 2. 标准化脚手架设计 (`trust-cloud-parent`)
所有微服务必须继承此 Parent，统一管理依赖版本，严禁私自引入异构 Jar 包。

### 2.1 核心模块划分
+ `trust-common-core`:
    - **精度控制**：全局 Jackson 配置，强制将 `BigDecimal` 序列化为 String，**彻底杜绝前端 JS 精度丢失**。
    - **异常拦截**：统一拦截 `BusinessException` 返回业务提示；拦截 `SQLException/NPE` 返回“系统繁忙”并打印 TraceID。
+ `trust-common-web`:
    - **交互铁律**：**全 POST + JSON**。严禁使用 GET/PUT，严禁使用 URL 参数（`@RequestParam`）。所有入参继承 `BaseRequest`，响应统一 `R<T>`。
    - **防护**：内置 XSS 过滤器与防重放拦截器。
+ `trust-common-db`:
    - 集成 MyBatis Plus，配置 `BlockAttackInnerInterceptor`（防止全表更新）。
    - 配置 **Flyway**，应用启动时自动执行数据库变更脚本。
+ `trust-common-security`:
    - 实现 **RBAC-D** 模型。
    - 提供 `@DataScope` 注解，MyBatis 拦截器自动拼接权限 SQL（如：只查本部门数据）。
+ `trust-common-audit`:
    - 审计日志切面（详见下文）。

## 3. 核心中台能力套件 (The Core)
### 3.1 业务流程与表单引擎 (BPM & Low-Code)
+ **场景**：综合运营平台的开户、划款、用印审批。
+ **Flowable 封装**：
    - 屏蔽 `RuntimeService`。提供 `WorkflowService` 接口 (`start`, `approve`, `reject`)。
    - **状态回调**：封装统一监听器，流程结束自动回调业务服务更新状态。
+ **动态表单**：
    - 前端采用 **FormCreate**。
    - 后端使用 PostgreSQL **JSONB** 存储表单数据，避免为每个审批单建物理表。

### 3.2 规则计算引擎 (Rule Engine)
+ **场景**：固收业绩报酬计算、量化风控指标（如：持仓占比限制）。
+ **LiteFlow 封装**：
    - 业务逻辑拆解为原子组件（`Component`）。
    - 规则链通过 JSON/XML 配置，支持**热刷新**，调整风控规则无需重启服务。

### 3.3 统一防腐层网关 (ACL Gateway)
+ **定位**：作为微服务集群与外部系统（恒生、融选）的唯一关口。
+ **架构设计**：
    - **接口标准化**：将恒生 O45 的 Socket/DB 交互封装为标准 REST/Dubbo 接口。
    - **Mock 能力**：ACL 层内置 Mock 开关。开发环境开启时，直接返回模拟数据，解耦外部依赖。

### 3.4 分布式并发与事务
+ **并发控制**：
    - `@DistLock(key="#dealId")`: 基于 Redisson 看门狗机制，防止重复下单。
    - `@Idempotent`: 防止前端重复提交。
+ **可靠消息 (最终一致性)**：
    - 封装 `ReliableProducer`。采用**本地消息表**模式，发送前落库，收到 Ack 后更新状态，定时任务补发，确保消息必达。

## 4. 统一审计与合规体系
### 4.1 全链路审计日志 (Audit Log)
+ **目标**：记录“谁、在何时、修改了什么（变更前 vs 变更后）”。
+ **实现**：
    - 注解：`@AuditLog(module="产品", action="改费率", bizId="#id")`。
    - **Diff 机制**：AOP 切面在 Update 前后分别对对象做快照，计算差异（如 `rate: 0.05 -> 0.08`），异步写入 `sys_audit_log`。
+ **要求**：核心业务增删改接口必须添加此注解。

### 4.2 全链路跟踪日志：
+ **目标**：记录数据流程流程
+ **TraceID**：集成 Skywalking 或 Sleuth，确保 TraceID 贯穿网关->服务->DB->MQ 全链路。
+ **格式**：JSON 格式输出（方便 ELK 解析），包含 `[AppName, TraceID, Level, Class, Message]`。

### 4.3 敏感数据保护
+ **配置加密**：DB 密码、Redis 密码严禁明文写在配置文件，必须使用 Jasypt 加密。
+ **脱敏**：日志中严禁打印身份证、银行卡号等明文，必须通过掩码处理。

## 5. 数据库治理与设计规范
### 5.1 建模与版本管理
+ **建模工具**：强制使用 **PDManer** 或 **Chiner**。
    - **单一数据源**：PDManer 工程文件提交 Git，作为数据库设计的唯一真理。禁止口头建表。
+ **版本管理**：使用 **Flyway**。
    - 变更脚本（`V1.0.1__desc.sql`）提交至代码库，严禁人工直连生产库执行 DDL。
+ **注释要求**：表和字段必须包含 Comment，CI 流水线自动检查，缺失则构建失败。

### 5.2 关键设计标准
+ **ID 规范**：强制使用 **Snowflake (雪花算法)** 生成 19 位 Long ID。禁止自增。
+ **金额规范**：强制使用 `DECIMAL(20, 6)`。禁止 Float/Double。
+ **管控平台**：部署 **Archery** 或 **Bytebase**，所有临时 SQL 操作需工单审批。

### 5.2 接口与交互规范
1. **统一响应结构 (**`R<T>`**)**

```json
{
  "code": 200,
  "msg": "success",
  "data": { ... },
  "traceId": "a1b2c3d4" // 关联 SkyWalking ID，方便排错
}
```

2. **异常处理**
    - `BusinessException` -> 返回业务提示（前端弹窗）。
    - `Exception` (空指针/SQL错误) -> 返回“系统繁忙”，控制台打印堆栈告警，**严禁**将 SQL 错误直接暴露给前端。
3. **精度处理**
    - 后端 `BigDecimal` -> 前端 `String`。在 Jackson 配置中强制全局序列化，防止前端 JS 丢失精度。

   统一POST + JSON，开发效率高，规范好执行，安全性也更好控制。url路径不能有参数。

## 6. 质量保障与自动化测试
### 6.1 自动化测试体系
+ **接口集成测试**：
    - 工具：**MeterSphere** (推荐) 或 YApi。
    - 流程：在 MeterSphere 中编排业务场景（登录->下单->审批）。CI 流水线每次构建后自动触发回归测试。
+ **单元测试**：
    - 工具：JUnit 5 + Mockito。
    - 要求：核心计算模块（利息、估值）覆盖率 > 60%。

### 6.2 研发流程管控 (DevOps)
+ **代码仓库**：Git Flow 分支管理。Merge Request 必须经 Tech Lead 评审。
+ **质量门禁**：集成 **Corax**或其他。Bug 数 > 0 或 核心代码覆盖率不达标，流水线直接阻断。
+ **发布规范**：K8s 配置 `preStop` 钩子（sleep 10s）确保优雅停机。
