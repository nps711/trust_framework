# 信托业务中台项目梳理（合并版）

## 1. 这两个文件本质上想做的一件事
建设一套**可复制、可治理、可审计**的信托业务技术中台研发体系：  
用统一脚手架（`trust-cloud-parent` / `trust-cloud-parent`）把底层复杂能力标准化封装，让业务团队只关注业务逻辑，同时通过强约束规范保障金融系统的稳定性、合规性和交付效率。

换句话说：  
不是只做“一个脚手架”，也不是只写“一份规范”，而是要落地一整套“**规范 + 平台能力 + 工程治理 + 交付流程**”的中台工程体系。

## 2. 两份文档的分工与关系
## 2.1 《技术中台建设规范白皮书》
- 定义“为什么做、做成什么样”。
- 给出架构原则、技术选型、合规红线、数据库规范、测试与发布标准。
- 强调金融场景的稳定性与审计要求（审计日志、TraceID、敏感数据保护、数据变更管控）。

## 2.2 《trust-cloud-parent 技术设计方案》
- 定义“具体怎么做”。
- 拆解脚手架模块（8+N），明确每个模块职责和依赖边界。
- 给出关键技术落地机制（上下文透传、精度控制、数据权限、自动配置、治理流程）。

## 2.3 合并结论
- 白皮书是“目标蓝图 + 治理标准”。
- 设计方案是“实现蓝图 + 模块拆解”。
- 二者合起来，目标是：**建立企业级中台研发基座，并形成可持续演进的统一工程秩序**。

## 3. 项目范围（要交付什么）
1. 统一研发底座
- `trust-cloud-parent` Root BOM + Parent 体系。
- 统一依赖版本、统一自动配置、统一基础能力接入方式。

2. 通用能力组件（脚手架模块）
- Core/Web/DB/Security/Log/Config/RPC/Redis/MQ/File 等标准模块。
- 开箱即用，按需引入，避免业务重复造轮子。

3. 核心中台能力
- 流程与表单（Flowable + 动态表单）。
- 规则计算（LiteFlow）。
- ACL 防腐层（外部系统标准化接入与隔离）。
- 分布式锁、幂等、可靠消息最终一致性。

4. 合规与可观测体系
- 审计日志（变更前后 Diff）。
- 全链路追踪（TraceID 贯通 Web/RPC/MQ/DB）。
- 敏感信息脱敏与配置加密。

5. 数据与接口治理
- 数据库建模、Flyway 版本化、SQL 变更管控。
- 统一响应结构、统一异常处理、统一 POST + JSON 交互规范。
- 金额与精度规范（BigDecimal/DECIMAL，前后端精度一致）。

6. 工程治理与发布机制
- 代码规范、MR 审核机制、脚手架发布权限与版本节奏。
- 自动化测试体系（集成测试 + 单测覆盖门槛）。
- CI/CD 质量门禁与违规阻断。

## 4. 落地路径（建议执行节奏）
1. 第1-2周：基座搭建
- 搭环境（K8s、PG、Apollo、XXL-JOB、测试平台）。
- 发布首版 Parent/BOM 与核心 Common 包。

2. 第3周：试点验证
- 选一个真实流程（如账户审批）跑通端到端：
  流程引擎 + 数据库版本化 + 统一接口规范 + 审计追踪。

3. 第4周：全面推广
- 发布开发手册、项目模板、接入清单。
- 以 CI 门禁 + 规范审查推动全团队统一落地。

## 5. 成功标准（验收口径）
1. 新业务服务可在短时间内按模板启动，并通过统一规范运行。
2. 关键能力不再在业务项目重复实现（锁、审计、权限、追踪、异常处理）。
3. 上线流程具备可审计、可回溯、可治理能力（日志、版本、审批链完整）。
4. 架构约束可被工具化执行（CI 阻断违规，而非口头约束）。

## 6. 一句话项目定义
这是一个“**把金融级研发能力产品化**”的项目：  
通过统一脚手架与治理体系，把“人治开发”升级为“标准化、自动化、可审计”的工业化研发模式。
